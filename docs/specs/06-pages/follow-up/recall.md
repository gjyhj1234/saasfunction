| 行为 | 说明 |
|------|------|
| 切换筛选项 | 切换后不自动查询，需点击 [查询] 按钮 |

## 状态机/流转

### 复诊任务流转

```
┌──────────────┐
│   pending    │ (待复诊 - 待复诊标签页)
└──┬───────────┘
   │
   │ 到达计划日期未完成
   ▼
┌──────────────┐
│   overdue    │ (逾期 - 逾期未复诊标签页)
└──┬──┬──┬─────┘
   │  │  │
   │  │  │ 记录失约
   │  │  ▼
   │  │ ┌──────────────┐
   │  │ │   no_show    │ (失约 - 失约记录标签页)
   │  │ └──┬───────────┘
   │  │    │ 重新预约
   │  │    ▼
   │  │  ┌──────────────┐
   │  │  │ rescheduled  │ (已改期) ──→ 生成新 pending 任务
   │  │  └──────────────┘
   │  │
   │  │ 发送提醒后患者回复/创建预约
   │  ▼
   │ ┌──────────────┐
   │ │  contacted   │ (已联系)
   │ └──┬───────────┘
   │    │ 患者就诊
   │    ▼
   ├──→┌──────────────┐
       │  completed   │ (已完成 - 从列表移除)
       └──────────────┘
```

### 提醒发送流转

```
┌──────────────┐
│   pending    │ (待发送)
└──────┬───────┘
       │ 发送
       ▼
┌──────────────┐
│    sent      │ (已发送)
└──────┬───────┘
       │
       ├─── 送达 ──→ ┌──────────────┐
       │              │  delivered   │ (已送达)
       │              └──────┬───────┘
       │                     │ 已读
       │                     ▼
       │              ┌──────────────┐
       │              │    read      │ (已读)
       │              └──────────────┘
       │
       └─── 失败 ──→ ┌──────────────┐
                      │   failed     │ (发送失败)
                      └──────────────┘
```

## 交互说明

### INT-01: 页面加载

| 步骤 | 行为 |
|------|------|
| 1 | 页面加载时根据 URL query 参数 `tab` 确定激活标签页; 默认 "待复诊" |
| 2 | 并行请求: 各标签页计数 + 当前标签页列表 + 医生下拉 |
| 3 | 标签页标题显示计数 Badge: "待复诊 (56)" / "逾期未复诊 (12)" / "失约记录 (8)" |
| 4 | 列表使用骨架屏作为加载占位 |
| 5 | 逾期行根据逾期天数显示不同背景色高亮 |

### INT-02: 标签页切换

| 步骤 | 行为 |
|------|------|
| 1 | 点击标签页切换; URL 参数同步更新 `tab=pending/overdue/no_show` |
| 2 | 重置筛选条件为当前标签页默认值 |
| 3 | 保留搜索关键词 (如有) |
| 4 | 请求对应标签页数据 |
| 5 | 切换时已选行清空 |

### INT-03: 发送提醒

| 步骤 | 行为 |
|------|------|
| 1 | 点击 [发送提醒] 打开提醒弹窗 |
| 2 | 默认渠道根据计划设定和患者联系方式确定: 有 WhatsApp → 默认 WhatsApp; 泰国号码 → 推荐 LINE |
| 3 | 显示自动生成的提醒内容模板，可手动修改 |
| 4 | 选择 "立即发送" 或 "定时发送" |
| 5 | 确认后调用发送接口 |
| 6 | 成功 → Toast "提醒已发送至 {channel}"; 更新提醒状态列 |
| 7 | 失败 → Toast (error) "提醒发送失败: {error_message}"; 可重试 |

### INT-04: 创建预约

| 步骤 | 行为 |
|------|------|
| 1 | 点击 [创建预约] 跳转至预约创建页 `/appointments/create` |
| 2 | URL 参数携带: `patient_id`, `doctor_id`, `task_id`, `source=recall` |
| 3 | 预约创建页自动预填患者、医生、建议日期 |
| 4 | 预约创建成功后, 对应任务自动关联预约 |

### INT-05: 标记已联系

| 步骤 | 行为 |
|------|------|
| 1 | 点击 [标记已联系] 立即更新联系状态 |
| 2 | 记录联系时间为当前时间 |
| 3 | 按钮变为 "已联系 ✅" 禁用状态 |
| 4 | 联系状态列更新为绿色 "已联系" Tag |

### INT-06: 记录失约原因

| 步骤 | 行为 |
|------|------|
| 1 | 点击 [记录原因] 打开失约原因弹窗 |
| 2 | 选择失约原因; 选 "其他" 时补充说明为必填 |
| 3 | 可选择是否需要重新预约 |
| 4 | 确认后保存; Toast "失约原因已记录" |
| 5 | 如勾选重新预约 → 提示 "是否立即创建预约？" → 跳转预约创建页 |

### INT-07: 批量操作

| 步骤 | 行为 |
|------|------|
| 1 | 勾选多条记录; 批量操作栏显示 "已选 {N} 条" |
| 2 | 批量发送提醒: 弹窗选择统一渠道或各自默认渠道 |
| 3 | 发送完成后显示结果汇总: "成功 {X} 条, 失败 {Y} 条" |
| 4 | 失败记录高亮显示; 支持重试失败项 |

## API 数据源

| 接口 | 方法 | 路径 | 请求参数 | 说明 |
|------|------|------|----------|------|
| 获取待复诊列表 | GET | `/api/v1/follow-up/recall/pending` | `keyword`, `days_range` (7/14/30/custom), `date_start`, `date_end`, `doctor_id`, `plan_type`, `reminder_status`, `sort_field`, `sort_order`, `page`, `page_size` | 获取即将到期的随访任务列表 |

| 获取失约记录列表 | GET | `/api/v1/follow-up/recall/no-show` | `keyword`, `date_start`, `date_end`, `doctor_id`, `no_show_reason`, `contact_status`, `sort_field`, `sort_order`, `page`, `page_size` | 获取失约记录列表 |
| 获取标签页计数 | GET | `/api/v1/follow-up/recall/counts` | - | 返回各标签页数据条数 |
| 发送提醒 | POST | `/api/v1/follow-up/recall/send-reminder` | `task_id`, `channel`, `content`, `send_time` | 发送复诊提醒 |
| 批量发送提醒 | POST | `/api/v1/follow-up/recall/batch-send-reminder` | `task_ids[]`, `channel`, `content` | 批量发送提醒 |
| 标记已联系 | PUT | `/api/v1/follow-up/recall/tasks/:id/contact` | - | 标记为已联系 |
| 批量标记已联系 | PUT | `/api/v1/follow-up/recall/batch-contact` | `task_ids[]` | 批量标记已联系 |
